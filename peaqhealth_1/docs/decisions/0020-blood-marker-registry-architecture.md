# ADR-0020: Blood marker registry as single source of truth

Date: 2026-05-01
Status: Foundation slice accepted — consumer cutover (Part 2) is a follow-up.

## Context

The audit at
[`docs/architecture/blood-markers-current-state.md`](../architecture/blood-markers-current-state.md)
documented severe drift across the blood-marker system:

- **Four parallel naming systems** for the same markers (DB column,
  parser key, TypeScript field, score-engine input) translated inline
  at four sites with no shared registry.
- **`BloodPanelData` is a 25-marker ceiling** in
  `apps/web/lib/user-context.ts`; everything outside that subset is
  invisible to converge / intervention / narrative logic — even though
  the `lab_results` table has 70 columns and the upload parser
  extracts ~50 fields.
- **Computed values silently dropped** (e.g. `ldlHdlRatio` is computed
  in the parser but never persisted to a column).
- **Database columns with no parser mapping** (e.g. `esr_mmhr` exists
  in the schema, the parser never extracts it).
- **Markers defined with non-existent column references** (e.g.
  `mpv_fl` defined in `apps/web/lib/markers/definitions.ts` but the
  table has no `mpv_fl` column).
- **`lab_history` materially narrower than `lab_results`** (27 vs 70
  columns), making trend charts dead UI for most markers.
- The Function Health 14-bug
  ([`docs/incidents/2026-05-01-function-health-14-bug.md`](../incidents/2026-05-01-function-health-14-bug.md))
  revealed parser fragility — per-format heuristics with the same
  text-flattening assumption silently failed on a different lab
  layout, persisting `14.0` for 31 of 31 markers.

The fix isn't another patch on the existing parser. It's collapsing
all four naming systems and the per-format parser logic into a single
registry that everything else generates from.

## Decision

### Single source of truth

[`apps/web/lib/blood/markerRegistry.ts`](../../apps/web/lib/blood/markerRegistry.ts) —
one row per marker. Every downstream concern reads from it:

- **Database column names** = registry IDs, exactly. The migration's
  column list is generated by hand against the registry, and the
  schema-sync test enforces it.
- **Parser output keys** = registry IDs, exactly. The OpenAI prompt
  is built from the registry's `synonyms`, `displayName`, and
  `expectedUnit`.
- **TypeScript marker-data type** = derived from the registry as
  `BloodMarkerData = { [K in (typeof BLOOD_MARKER_REGISTRY)[number]['id']]?: number | null }`.
- **UI dynamic-route validation** uses `isKnownMarkerId(slug)` — 404
  if not in registry.
- **Scoring inputs** read column names from the registry, not
  hardcoded strings. (Cutover is a follow-up; see § Out of scope.)
- **Save-time validation** uses `validRange` to reject parsing
  artifacts.

Adding a marker is **two changes**: a row in the registry plus a
column in a generated migration. The schema-sync test fails if
either is missing.

### Schema

Migration: `supabase/migrations/20260501_blood_marker_reset.sql`.

- `DROP TABLE IF EXISTS lab_history CASCADE;`
- `DROP TABLE IF EXISTS lab_results CASCADE;` — the CASCADE
  intentionally removes the `score_snapshots` rows that referenced
  it (which include the corrupt 14-bug data, see incident doc).
  The app is not in production.
- `CREATE TABLE blood_results` — one row per uploaded test
  (NOT per-user-upserted, the old failure mode). Includes 98
  marker columns from the registry + parser metadata
  (`source_lab`, `parser_used`, `parse_confidence`, `raw_pdf_path`,
  `collected_at`).
- `CREATE TABLE blood_marker_confidence` — one row per
  `(blood_result_id, marker_id)` with extracted confidence,
  raw OCR text, and a `was_computed` flag. Lets us debug
  parser quality without re-asking the user to re-upload — the
  forensic surface the 14-bug incident asked for.

The schema-sync test
[`apps/web/lib/blood/__tests__/registry-schema-sync.test.ts`](../../apps/web/lib/blood/__tests__/registry-schema-sync.test.ts)
parses the migration's `CREATE TABLE blood_results` body, extracts
the column list, and asserts:

1. Every registry id has a matching column.
2. Every non-reserved column corresponds to a registry id.
3. Registry ids are unique.
4. Every derived marker references valid operand ids.
5. Every marker has at least one synonym.
6. Every `validRange` is well-formed (`min < max`).

That's the "no drift possible" guarantee. The build fails if the two
diverge.

### Parser strategy

Single OpenAI vision call. No per-lab branching. The prompt is built
from the registry: target IDs, display names, synonyms, expected
units. The model returns JSON keyed by registry ID; the parser
validates against the registry's `validRange` and computes derived
markers when their operands are present.

[`apps/web/lib/blood/parser.ts`](../../apps/web/lib/blood/parser.ts) —
production parser. Renders PDF pages to PNG via `unpdf` (GPT-4o
vision can't take PDFs directly via `image_url`), sends them to
`gpt-4o`, gets JSON back, hands to the validation step.

[`apps/web/lib/blood/validate.ts`](../../apps/web/lib/blood/validate.ts) —
pure post-OpenAI logic, split out so unit tests can run without the
SDK. Three concerns:

1. **Range validation** — every extracted value is checked against the
   marker's `validRange`. Out-of-range → marker becomes `null`,
   warning emitted. Catches the LDL=14 / glucose=14000 class of
   layout artifacts at the value level.
2. **Derived computation** — for any marker with `derivedFrom`, if
   the operand markers are present and the derived value wasn't
   already extracted, compute it. `formula: "divide" | "percentage"
   | "ratio"`. Computed values are tagged `wasComputed: true` and
   their `rawExtractedText` becomes `[computed from <operands>]` so
   downstream consumers can distinguish.
3. **Uniform-value guard** — if ≥ 60 % of extracted (non-computed)
   markers share one value (with at least 5 markers populated), the
   parser **throws** rather than persists. This is the 14-bug
   backstop at the parser layer. Mirrors the same threshold as the
   write-time guard from
   [PR-251](https://github.com/drmemes1/peaqhealth/pull/251)
   (`apps/web/lib/labs/uniform-value-guard.ts`); both layers will
   coexist short-term, and the write-time guard becomes redundant
   once this parser replaces the existing upload route.

### Validation strategy

`validRange` bounds in the registry are **NOT clinical reference
ranges**. They are "is this a plausible human value or a parsing
artifact" bounds — generous, with the goal of catching obvious
extraction errors (LDL = 14, glucose = 14000) and not rejecting
any plausible value.

Clinical reference ranges live separately (in marker-content / UI
threshold bands) and depend on age / sex / context. That's a
different layer.

### Derived markers

Stored when extracted (lab printed it explicitly), computed from
operands when not. Six derived markers in the v1 registry:

| Marker | Formula | Operands |
|---|---|---|
| `total_chol_hdl_ratio` | divide | `total_cholesterol_mgdl`, `hdl_mgdl` |
| `bun_creatinine_ratio` | divide | `bun_mgdl`, `creatinine_mgdl` |
| `albumin_globulin_ratio` | divide | `albumin_gdl`, `globulin_gdl` |
| `iron_saturation_percent` | percentage | `iron_ugdl`, `iron_binding_capacity_ugdl` |
| `psa_free_percent` | percentage | `psa_free_ngml`, `psa_total_ngml` |
| `omega_6_3_ratio` | ratio | `omega6_total_percent`, `omega3_total_percent` |
| `arachidonic_epa_ratio` | ratio | `omega6_arachidonic_acid_percent`, `omega3_epa_percent` |

Computed values are themselves validated against the derived
marker's `validRange`. If the computation produces something
implausible (e.g. operands that explode the ratio), the marker
stays `null` and a warning is emitted.

### Tier discipline

`tier: "standard" | "pro"`. Pro-tier markers (`requiresAdvancedTest:
true` for NMR LipoProfile, omega index, autoantibodies, heavy
metals, PSA panel, advanced inflammation) are present in the registry
and schema but expected to be null for users on standard panels. The
UI can filter by tier when rendering.

### Categorical markers — deferred

LDL Pattern (A or B) and ABO Group are categorical, not numeric. The
v1 schema is `numeric` everywhere. Adding categoricals is a
follow-up — the registry's `BloodMarker` type will need a
discriminator. Out of scope for v1.

## Out of scope (follow-up PR)

The brief described a full architectural reset including consumer
cutover. This slice ships the **foundation** — registry, migration,
parser, validation, schema-sync test. The cutover is intentionally
deferred to a separate PR because:

- 20+ consumer files reference `lab_results` directly (audited).
- `BloodPanelData` is consumed across converge / intervention /
  narrative logic; the safest path is a `bloodPanelDataFromBloodResults`
  adapter that translates at the read boundary, then per-route
  cleanup.
- The score engine has its own naming (`BloodInputs` with
  camelCase + unit suffix); a translation map at the engine
  boundary is needed.
- Reviewing the registry shape and migration columns BEFORE
  committing the team to migrating 20+ files is the cheaper
  ordering.

Items deferred to the consumer-cutover follow-up:

- Save route rewrite (`apps/web/app/api/labs/save/route.ts`) — wire
  to the new parser, write to `blood_results` + `blood_marker_confidence`.
- Adapter: `bloodPanelDataFromBloodResults()` that produces the
  existing 25-field `BloodPanelData` shape from a `blood_results`
  row, so converge / intervention / narrative consumers don't break
  on the schema change.
- `/dashboard/blood/[marker]` route validation against the registry
  (404 for unknown ids, render from registry definition).
- Deletion of `apps/web/lib/markers/definitions.ts`,
  `apps/web/lib/blood/marker-content.ts` (the duplicate registries),
  and `BloodPanelData` / `BloodMarkers` types (replaced by
  registry-derived types).
- Removal of `lab_history` references in code (the table is gone;
  trend infra rebuilds against `blood_results` ordered by
  `collected_at`).
- Score engine input translation.

## What this PR does NOT touch

- Sleep, oral, lifestyle scoring or schema.
- Junction integration code.
- Wearable code.
- Auth / onboarding.
- Existing `lab_results` data (none in production matters; the
  `score_snapshots` corruption from the 14-bug incident is part of
  the CASCADE removal).
- Unit conversion (parser stores units as found; `expectedUnit` in
  the registry is informational for now).
- Re-processing of past blood_results when registry changes (when
  you add a marker, you need to re-upload PDFs to capture it).
- Raw PDF storage workflow (`raw_pdf_path` column exists; storage
  upload is a separate piece).
